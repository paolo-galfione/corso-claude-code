## SLIDE 2: Cosa Sono gli Hooks

**Contenuto slide**:
```
COSA SONO GLI HOOKS

Trigger automatici su eventi Claude Code

┌─────────────────────────────────────────┐
│  EVENTO          →    AZIONE            │
├─────────────────────────────────────────┤
│  Pre-commit      →    Esegui test       │
│  Post-edit       →    Formatta codice   │
│  On-error        →    Log dettagliato   │
│  Pre-push        →    Verifica build    │
└─────────────────────────────────────────┘

Se l'azione fallisce, l'operazione si blocca
```

**Testo da leggere**:
"Un hook è: quando succede X, fai Y. Se Y fallisce, blocca l'operazione. Questo garantisce che certe condizioni siano sempre verificate. Non potete fare commit se i test falliscono. Non potete pushare se la build è rotta."

---

## SLIDE 3: Tipi di Hooks

**Contenuto slide**:
```
TIPI DI HOOKS

PreToolCall
└─ Prima che Claude esegua un tool
└─ Es: verifica permessi, log azione

PostToolCall
└─ Dopo che Claude ha eseguito un tool
└─ Es: formattazione, validazione output

Notification
└─ Eventi informativi
└─ Es: log, metriche, alert

Stop
└─ Termina l'esecuzione se condizione vera
└─ Es: troppi errori, file proibito modificato
```

**Testo da leggere**:
"Ci sono diversi tipi di hooks. Pre e Post per azioni prima e dopo i tool. Notification per logging e alert. Stop per bloccare l'esecuzione in casi critici. Ogni tipo ha il suo uso specifico."

```
CONFIGURAZIONE

```

**Testo da leggere**:
"Gli hooks si configurano in settings.json. Ogni hook ha un matcher che specifica su quali tool scatta, e un command che viene eseguito. Nell'esempio: prima di scrivere logghiamo, dopo formattiamo automaticamente."

## DEMO 1: Hook Post-Edit per Linting

**Preparazione** - file da avere pronti nel progetto:

1. `package.json`:

```json
{
  "name": "demo-hooks",
  "version": "1.0.0",
  "scripts": {
    "lint": "eslint .",
    "lint:fix": "eslint --fix ."
  },
  "devDependencies": {
    "eslint": "^8.57.0"
  }
}
```

2. `.eslintrc.json`:

```json
{
  "env": { "node": true, "es2021": true },
  "rules": {
    "quotes": ["error", "single"],
    "semi": ["error", "always"],
    "indent": ["error", 2]
  }
}
```

3. `src/api.js` (codice iniziale formattato correttamente):

```javascript
const express = require('express');
const router = express.Router();

const users = [
  { id: 1, name: 'Alice' },
  { id: 2, name: 'Bob' },
  { id: 3, name: 'Charlie' }
];

router.get('/users', (req, res) => {
  res.json(users);
});

module.exports = router;
```

**Cosa mostrare**:

1. Prompt 1 - Far creare l'hook a Claude:

```
Configura un hook PostToolCall in .claude/settings.json che esegua
ESLint --fix automaticamente su ogni file che modifichi.
L'hook deve attivarsi sui tool Write e Edit.
```

2. Claude crea `.claude/settings.json`:

```json
{
  "hooks": {
    "PostToolCall": [
      {
        "matcher": "Write|Edit",
        "command": "npx eslint --fix \"$CLAUDE_FILE_PATH\" 2>/dev/null || true"
      }
    ]
  }
}
```

3. Prompt 2 - Testare l'hook:

```
Modifica src/api.js: aggiungi una funzione getUserById(id)
che trova un utente per ID e un endpoint GET /users/:id.
Usa double quotes per le stringhe e niente semicoloni a fine riga.
```

4. Claude scrive il codice con formattazione sbagliata (double quotes, no semicoloni)
5. L'hook PostToolCall si attiva automaticamente dopo la scrittura
6. ESLint --fix corregge: quotes singole, semicoloni aggiunti
7. Il file finale risulta formattato secondo le regole:

```javascript
const express = require('express');
const router = express.Router();

const users = [
  { id: 1, name: 'Alice' },
  { id: 2, name: 'Bob' },
  { id: 3, name: 'Charlie' }
];

router.get('/users', (req, res) => {
  res.json(users);
});

function getUserById(id) {
  return users.find(u => u.id === id);
}

router.get('/users/:id', (req, res) => {
  const user = getUserById(req.params.id);
  if (!user) return res.status(404).json({ error: 'Not found' });
  res.json(user);
});

module.exports = router;
```

**Testo da leggere durante la demo**:
"Prima facciamo creare l'hook a Claude. Guardate il settings.json: PostToolCall, matcher Write e Edit, comando ESLint --fix. Ora testiamolo: chiediamo a Claude di scrivere codice con formattazione sbagliata, double quotes e senza semicoloni. Claude scrive, il file appare, e immediatamente l'hook scatta. ESLint corregge automaticamente. Il codice finale rispetta le nostre regole senza che nessuno debba pensarci."

## SLIDE 5: Hook per Test Automatici

**Contenuto slide**:
```
HOOK PER TEST AUTOMATICI

Dopo ogni modifica a file .js in /routes, /services, /repositories:
esegui i test correlati

{
  "PostToolCall": [
    {
      "matcher": "Write|Edit",
      "command": "./scripts/run-related-tests.sh $FILE"
    }
  ]
}

Script run-related-tests.sh:
- Identifica test correlati al file
- Esegue solo quei test
- Feedback rapido senza full suite
```

**Testo da leggere**:
"Questo hook esegue test correlati dopo ogni modifica. Non la full suite, solo i test rilevanti. Feedback rapido: se hai rotto qualcosa, lo sai subito. L'agente può correggere prima di procedere."

## DEMO 2: Hook con Validazione

**Preparazione** - file da avere pronti nel progetto:

1. `config/production.json` (file sensibile da proteggere):

```json
{
  "database": {
    "host": "prod-db.internal",
    "port": 5432,
    "password": "s3cret_pr0d_password"
  },
  "api_key": "sk-prod-xxxxxxxxxxxx"
}
```

2. `migrations/001_init.sql`:

```sql
-- Migration 001: Schema iniziale
CREATE TABLE users (id INTEGER PRIMARY KEY, name TEXT);
```

**Cosa mostrare**:

1. Prompt 1 - Far creare l'hook a Claude:

```
Aggiungi un hook PreToolCall in .claude/settings.json che impedisca
la modifica di file sensibili. Crea uno script scripts/check-protected-files.js
che blocchi modifiche a: config/production.json, la directory migrations/, e .env.
Se il file è protetto, lo script deve uscire con exit code 1
e un messaggio di errore chiaro.
```

2. Claude aggiorna `.claude/settings.json` aggiungendo il PreToolCall:

```json
{
  "hooks": {
    "PostToolCall": [
      {
        "matcher": "Write|Edit",
        "command": "npx eslint --fix \"$CLAUDE_FILE_PATH\" 2>/dev/null || true"
      }
    ],
    "PreToolCall": [
      {
        "matcher": "Write|Edit",
        "command": "node ./scripts/check-protected-files.js \"$CLAUDE_FILE_PATH\""
      }
    ]
  }
}
```

3. Claude crea `scripts/check-protected-files.js`:

```javascript
const protectedPaths = [
  'config/production.json',
  'migrations/',
  '.env'
];

const file = process.argv[2] || '';
const isProtected = protectedPaths.some(p => file.includes(p));

if (isProtected) {
  console.error(`BLOCCATO: ${file} è un file protetto`);
  console.error('I file di produzione e le migration non possono essere modificati.');
  process.exit(1);
}
```

4. Prompt 2 - Testare l'hook:

```
Aggiorna la configurazione del database in config/production.json
per usare il nuovo host: prod-db-v2.internal
```

5. Claude tenta di modificare `config/production.json`
6. L'hook PreToolCall scatta PRIMA della modifica
7. Lo script trova il file nella lista protetta e lo blocca:

```
BLOCCATO: config/production.json è un file protetto
I file di produzione e le migration non possono essere modificati.
```

8. Il file NON viene modificato - l'operazione è cancellata
9. Claude risponde spiegando che non può modificare il file per via della protezione

**Testo da leggere durante la demo**:
"Ora facciamo creare un hook di sicurezza. Chiediamo a Claude di configurare un PreToolCall che blocchi i file sensibili. Guardate: crea lo script e aggiorna settings.json. Ora testiamolo: chiediamo di modificare config/production.json. L'hook scatta PRIMA che Claude possa toccare il file, lo script esce con errore, l'operazione viene cancellata. Il file resta intatto. Sicurezza enforced automaticamente."

## SLIDE 7: Hooks e Auto-Correzione

**Contenuto slide**:
```
HOOKS E AUTO-CORREZIONE

Hook fallisce → Claude riceve errore → Corregge

┌─────────────────────────────────────────┐
│  1. Claude modifica file                │
│  2. Hook lint fallisce                  │
│  3. Claude vede errore ESLint           │
│  4. Claude corregge il problema         │
│  5. Hook passa                          │
│  6. Operazione completa                 │
└─────────────────────────────────────────┘

Il loop agentico include gli hooks
```

**Testo da leggere**:
"Quando un hook fallisce, Claude vede l'errore e può correggerlo. Non è un blocco definitivo, è feedback nel loop agentico. Claude scrive codice con errore di lint, l'hook segnala, Claude corregge, l'hook passa. Automazione e auto-correzione insieme."

## DEMO 3: Hook che Innesca Auto-Correzione

**Preparazione**:

1. Aggiungere la regola `no-var` a `.eslintrc.json`:

```json
{
  "env": { "node": true, "es2021": true },
  "rules": {
    "quotes": ["error", "single"],
    "semi": ["error", "always"],
    "indent": ["error", 2],
    "no-var": ["error"]
  }
}
```

**Cosa mostrare**:

1. Prompt 1 - Modificare l'hook per non auto-correggere:

```
Modifica l'hook PostToolCall in .claude/settings.json:
rimuovi il --fix e il || true dal comando ESLint.
Voglio che l'hook SEGNALI gli errori senza correggerli,
così se c'è un problema di lint tu lo vedi e lo correggi tu stesso.
```

2. Claude modifica l'hook PostToolCall:

```json
{
  "hooks": {
    "PostToolCall": [
      {
        "matcher": "Write|Edit",
        "command": "npx eslint \"$CLAUDE_FILE_PATH\" --max-warnings 0"
      }
    ],
    "PreToolCall": [
      {
        "matcher": "Write|Edit",
        "command": "node ./scripts/check-protected-files.js \"$CLAUDE_FILE_PATH\""
      }
    ]
  }
}
```

3. Prompt 2 - Testare il loop di auto-correzione:

```
Modifica src/api.js: aggiungi una funzione getAllUserNames()
che estrae tutti i nomi degli utenti dall'array.
Usa lo stile JavaScript classico con var.
```

4. Claude scrive il codice usando `var` (come richiesto):

```javascript
function getAllUserNames() {
  var names = users.map(u => u.name);
  return names;
}
```

5. L'hook PostToolCall si attiva, ESLint analizza il file
6. ESLint trova la violazione e l'hook FALLISCE:

```
src/api.js
  18:3  error  Unexpected var, use let or const instead  no-var

✖ 1 problem (1 error, 0 warnings)
```

7. Claude riceve l'errore nel contesto, lo legge, capisce che `var` non è ammesso
8. Claude auto-corregge: modifica il file cambiando `var` in `const`

```javascript
function getAllUserNames() {
  const names = users.map(u => u.name);
  return names;
}
```

9. L'hook PostToolCall si riattiva, ESLint analizza di nuovo
10. Nessun errore: l'hook PASSA, operazione completata

11. Evidenziare il loop completo:
    **Write → Hook fails → Claude reads error → Claude corrects → Write → Hook passes**

**Testo da leggere durante la demo**:
"Questa è la demo cruciale. Cambiamo l'hook: ora ESLint segnala ma non corregge. Chiediamo a Claude di usare 'var', che le nostre regole vietano. Claude scrive, l'hook scatta, ESLint trova l'errore, l'hook fallisce. Ora guardate: Claude riceve l'errore, lo legge, capisce che var non è ammesso, corregge da solo cambiando var in const, salva di nuovo. L'hook riscatta, questa volta passa. Questo è il loop agentico: l'hook dà feedback, Claude corregge, l'hook verifica. Automazione e intelligenza insieme."

## SLIDE 8: Best Practice

**Contenuto slide**:
```
BEST PRACTICE

✓ Hooks veloci (< 5 secondi ideale)
✓ Messaggi di errore chiari
✓ Fail fast: blocca subito se critico
✓ Log per debugging
✓ Test degli hooks stessi

✗ Hooks che richiedono minuti
✗ Errori criptici
✗ Troppi hooks sovrapposti
✗ Hooks che modificano silenziosamente
```

**Testo da leggere**:
"Gli hooks devono essere veloci, altrimenti rallentano il flusso. Messaggi chiari così Claude capisce cosa correggere. Non sovrapponete troppi hooks. E testate gli hooks: un hook rotto blocca tutto il lavoro."
