# Screencast 2.2: Validazione e Criteri di Accettazione

**Durata**: max 10 minuti
**Modulo**: 2 - Il Product Owner
**Scopo**: Usare Claude per definire criteri di qualità misurabili

---

## SLIDE 1: Titolo

**Contenuto slide**:
```
VALIDAZIONE E CRITERI DI ACCETTAZIONE

Definire il successo prima di iniziare
```

**Testo da leggere**:
"Abbiamo analizzato il problema e scelto l'approccio. Ora dobbiamo definire cosa significa 'fatto'. Criteri chiari e misurabili che permettano a tutti, umani e agenti, di sapere quando il lavoro è completo."

---

## SLIDE 2: Perché i Criteri di Accettazione

**Contenuto slide**:
```
PERCHÉ SERVONO CRITERI CHIARI

Per gli umani:
- Allineamento su cosa significa "finito"
- Base per la review
- Misura oggettiva del progresso

Per gli agenti:
- Obiettivo verificabile autonomamente
- Criterio di stop del loop agentico
- Base per auto-correzione
```

**Testo da leggere**:
"I criteri di accettazione servono a umani e agenti. Per noi sono allineamento e misura del progresso. Per gli agenti sono ancora più importanti: definiscono quando fermarsi e cosa verificare. Senza criteri chiari, l'agente non sa quando ha finito."

---

## SLIDE 3: Anatomia di un Buon Criterio

**Contenuto slide**:
```
ANATOMIA DI UN BUON CRITERIO

┌─────────────────────────────────────────┐
│  ✗ "Il codice deve essere pulito"       │
│                                         │
│  ✓ "Ogni file contiene al massimo       │
│     una classe/modulo"                  │
│                                         │
│  ✓ "Nessuna funzione supera 50 righe"   │
│                                         │
│  ✓ "Copertura test > 80% sui Service"   │
└─────────────────────────────────────────┘

Misurabile → Verificabile → Automatizzabile
```

**Testo da leggere**:
"Un buon criterio è misurabile. 'Codice pulito' non significa nulla. 'Un file, una responsabilità' è verificabile. 'Copertura test sopra 80%' è automatizzabile. Se un criterio non può essere verificato automaticamente, probabilmente è troppo vago."

---

## DEMO 1: Generare Criteri con Claude

**Cosa mostrare**:

1. Aprire Claude Code
2. Prompt: "Basandoti sull'analisi del backend legacy che abbiamo fatto, genera i criteri di accettazione per la Fase 2 del refactoring: separazione in architettura 3-tier. I criteri devono essere misurabili e verificabili automaticamente."

**Claude dovrebbe produrre criteri come**:
- Struttura file: API in /routes, Service in /services, Repository in /repositories
- Ogni route handler chiama solo metodi del Service layer
- Nessuna query SQL nei file /routes
- Ogni Service ha unit test con mock del Repository
- Test esistenti (baseline) continuano a passare

**Testo da leggere durante la demo**:
"Chiediamo a Claude di generare criteri misurabili per la Fase 2. Nota come ogni criterio può essere verificato: o con un test, o con un'analisi statica del codice, o guardando la struttura delle cartelle."

---

## SLIDE 4: Criteri per Ogni Fase

**Contenuto slide**:
```
CRITERI PER FASE

Fase 0 - Test Baseline
└─ "Tutti gli endpoint hanno almeno un test E2E"
└─ "I test catturano il comportamento attuale"

Fase 1 - Bug Fix
└─ "Zero SQL injection (prepared statements)"
└─ "Filtri funzionano correttamente"
└─ "N+1 query eliminati"

Fase 2 - Separazione 3-tier
└─ "Zero query SQL fuori da /repositories"
└─ "Zero logica di business in /routes"

...continua
```

**Testo da leggere**:
"Ogni fase ha i suoi criteri specifici. La Fase 0 verifica che abbiamo test di baseline. La Fase 1 fixa i bug critici. La Fase 2 verifica la separazione. Ogni fase è un rilascio: se i criteri passano, possiamo rilasciare."

## DEMO 2: Validare Documentazione Esistente

**Cosa mostrare**:

1. Mostrare un documento di requisiti (anche fittizio/semplice)
2. Prompt: "Ecco il documento dei requisiti per il refactoring. Analizzalo e dimmi: 1) Ci sono ambiguità? 2) Mancano informazioni critiche? 3) I criteri di accettazione sono verificabili?"

**Claude dovrebbe**:
- Identificare ambiguità nel documento
- Segnalare informazioni mancanti
- Evidenziare criteri non misurabili
- Suggerire miglioramenti specifici

**Testo da leggere durante la demo**:
"Claude può anche validare documentazione esistente. Gli passiamo un documento e chiediamo di trovare problemi: ambiguità, informazioni mancanti, criteri vaghi. È come avere un reviewer sempre disponibile."

## DEMO 3: Generare il Piano Completo

**Cosa mostrare**:

1. Prompt: "Genera un piano di rilascio completo per il refactoring del backend Northwind. Per ogni fase indica: obiettivo, criteri di accettazione verificabili, dipendenze dalla fase precedente, rischi e mitigazioni."

**Claude dovrebbe produrre**:
- Piano strutturato per tutte le fasi
- Criteri specifici e misurabili per ogni fase
- Dipendenze chiare (Fase 2 richiede Fase 1 completata)
- Rischi identificati con mitigazioni

2. Mostrare come il PO può iterare: "Per la Fase 1, aggiungi criteri specifici per i bug di sicurezza"

**Testo da leggere durante la demo**:
"Claude genera il piano completo, poi possiamo iterare su parti specifiche. Vogliamo più dettaglio sui bug di sicurezza? Chiediamo. Il piano evolve attraverso la conversazione."
