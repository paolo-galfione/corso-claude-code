## SLIDE 2: Dal Piano alle Specifiche

**Contenuto slide**:
```
DAL PIANO ALLE SPECIFICHE

Piano (cosa fare):
└─ "Fase 1: Separare in 3 tier"

Specifica (come farlo):
└─ "Per l'endpoint GET /customers:
    1. Crea customerService.js
    2. Sposta la logica di filtro
    3. Crea customerRepository.js
    4. Sposta le query SQL
    5. Aggiorna route per usare service
    6. Verifica con test"

Il piano dice COSA, la specifica dice COME
```

**Testo da leggere**:
"Il piano dice cosa fare a livello strategico. La specifica dice come farlo operativamente. È il livello di dettaglio che permette a qualcuno, umano o agente, di eseguire senza dover prendere decisioni architetturali."

## SLIDE 3: Struttura di una Specifica

**Contenuto slide**:
```
STRUTTURA DI UNA SPECIFICA

# Specifica: [Nome]

## Contesto
Perché stiamo facendo questo

## Prerequisiti
Cosa deve essere già completato

## Obiettivo
Risultato atteso

## Passi
1. Azione specifica
2. Azione specifica
...

## Criteri di Accettazione
□ Verifica 1
□ Verifica 2

## Note
Edge case, attenzioni particolari
```

**Testo da leggere**:
"Una specifica ha struttura chiara. Contesto per capire il perché. Prerequisiti per sapere cosa serve prima. Obiettivo per sapere dove arrivare. Passi dettagliati. Criteri di accettazione per verificare. Note per casi particolari."

## DEMO 2: Specifica con Criteri di Verifica

**Cosa mostrare**:

1. Chiedere di aggiungere criteri di verifica specifici:

```
Per ogni passo della specifica, aggiungi il criterio
di verifica che l'agente deve controllare prima di
procedere al passo successivo.

Esempio:
Passo: "Crea customerService.js"
Verifica: "File esiste in /services/, contiene classe
          CustomerService con metodi getAll, getById"
```

2. Claude aggiorna la specifica con verifiche per ogni passo

**Testo da leggere durante la demo**:
"Aggiungiamo criteri di verifica per ogni passo. L'agente non procede al passo 2 finché la verifica del passo 1 non passa. Questo previene errori a catena: se qualcosa va storto, ci fermiamo subito."


**Testo da leggere**:
"Documentate le decisioni nella specifica. Perché quest'ordine? Come gestiamo gli errori? Quali naming convention? Queste decisioni evitano che ogni sviluppatore o agente faccia scelte diverse."

## SLIDE 8: Best Practice

**Contenuto slide**:
```
BEST PRACTICE

✓ Una specifica per task atomico
✓ Criteri di verifica per ogni passo
✓ Dipendenze esplicite
✓ Template per task ripetitivi
✓ Decisioni documentate

✗ Specifiche troppo ampie
✗ Passi vaghi ("sistema il codice")
✗ Assunzioni implicite
✗ Dipendenze nascoste
```

**Testo da leggere**:
"Specifiche atomiche, verificabili passo per passo. Dipendenze esplicite, template per consistenza. Evitate specifiche troppo ampie che diventano impossibili da seguire."
